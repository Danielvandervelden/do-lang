# Consolidate Claude/Codex Skill Trees

## Vision

Consolidate the duplicated Claude and Codex skill, agent, and reference markdown trees in do-lang into a single canonical source in `skills/_source/` (a new template directory) with a build step that generates platform-specific variants into `skills/do/` (Claude-flavored) and `skills/codex/` (Codex-flavored) at pack/publish time. Both generated directories ship inside the npm tarball as static artifacts -- `install.cjs` copies from them at postinstall time, requiring no runtime generation and no changes to its copy logic.

Currently the repo contains 68 paired files (47 reference pairs, 13 entry skill pairs, 8 agent pairs). Of these, 23 are byte-identical and can be directly shared. The remaining 45 diverge in predictable, mechanical ways: install path substitution (`~/.claude/commands/do/` vs `~/.codex/skills/do/`), agent name prefix rewriting (`do-` vs `codex-`), Agent() JS spawn blocks vs natural-language spawn directives, and minor prose wording differences. A template system with ERB-style delimiters (`<%= VAR %>` for substitution, `<% if platform %>...<% endif %>` for conditionals) can handle all observed divergence patterns. ERB-style delimiters are chosen because existing scaffold/reference files already use `{{...}}` Mustache-style placeholders (e.g., `{{TITLE}}`, `{{VISION}}`) as intentional content — using the same delimiter would cause collisions.

This eliminates the maintenance cost of keeping these files in sync across two platform targets, benefiting the sole maintainer (Daniel) by making skill authoring and updates a single-file operation instead of a copy-and-adapt workflow.

## Target Users

- **Daniel (sole maintainer)** -- Expert-level developer who maintains the do-lang project. Edits canonical skill/agent files directly and runs the build step during development and release cycles. The only person who authors, modifies, and publishes skills.

## Non-Goals

- No new skills or behavior changes -- this is structural consolidation only
- No third platform targets -- Claude + Codex only (no Cursor, Windsurf, etc.)
- No changes to skill semantics or runtime behavior
- No install UX enhancements -- "smart detection" (auto-detecting which CLIs are installed and skipping unavailable targets) is a separate concern from deduplication and will not be addressed in this project

## Success Criteria

1. **Single source of truth** -- each skill, agent, and reference exists in exactly one canonical file (in `skills/_source/` or `agents/_source/`) with template markers; platform variants are generated by the build step (no hand-maintained duplicates)
2. **Behavioral parity** -- `installClaudeCode()` and `installCodex()` produce bit-identical output to the v1.19.0 baseline snapshot (verified by SHA-256 hash comparison)
3. **Measurably fewer hand-maintained files** -- eliminate 68 hand-maintained duplicate files (47 reference pairs + 13 entry skill pairs + 8 agent pairs). Generated output (`skills/do/`, `skills/codex/`, `agents/do-*.md`, `agents/codex-*.md`) still exists at pack time but is not committed to git.
4. **Build step works** -- `npm run prepack` (or equivalent) regenerates all platform variants from canonical sources without errors

## Constraints

- **Backward-compatible install** -- existing consumers running `npm install -g` must not break; postinstall output paths must remain unchanged
- **Override behavior** -- install flow should override existing installation
- **Platform prompt preserved** -- preserve the "do you want to install to claude, codex or both" prompt
- **No timeline pressure** -- personal project, quality over speed
- **Tech stack** -- Node.js 18+, CommonJS modules (.cjs), npm package
- **Canonical source location** -- `skills/_source/` is the canonical template source of truth. Template markers (ERB-style: `<%= PLATFORM_PATH %>`, `<% if claude %>...<% endif %>`) are added to files in this directory. Both `skills/do/` (Claude-flavored) and `skills/codex/` (Codex-flavored) become generated output directories, populated by the build step. This avoids the problem of template markers leaking into installed files: `install.cjs` copies from `skills/do/` and `skills/codex/` which contain fully resolved (marker-free) content. The author edits files in `skills/_source/` and the build step produces both platform variants.
- **Build timing: pack-time generation** -- Both platform variants are generated before `npm pack`/`npm publish` via a `prepack` script. The npm tarball ships with fully-generated `skills/do/` and `skills/codex/`. This means `install.cjs` continues to work unchanged (it copies from directories that already exist in the tarball with resolved content). No runtime generation at postinstall. This avoids: (a) requiring the template engine at install time, (b) slowing down consumer installs, (c) breaking in restricted environments.
- **scripts/ directory** -- `scripts/` lives only in `skills/_source/` (the canonical location). The build step copies `scripts/` into `skills/do/` as-is (no template processing -- scripts are .cjs/.js, not markdown). The build step does NOT copy scripts into `skills/codex/`. Instead, `install.cjs` already handles Codex scripts: `installCodex()` explicitly copies `skills/do/scripts/` into the Codex install target. This behavior is preserved unchanged.
- **install.cjs unchanged** -- Both `skills/do/` and `skills/codex/` must exist in the tarball at install time with fully resolved content. Since generation happens at prepack, this is guaranteed. No changes to `install.cjs` copy logic are needed in any phase. The only change to `install.cjs` is adding a `require.main === module` guard so the install functions can be exported for baseline testing (Phase 00).
- **Dev workflow after consolidation** -- After generated dirs stop being committed, a fresh checkout requires `npm run build` (or equivalent) before `npm test` or local `npm install -g .` will work (since `skills/do/` and `agents/do-*.md` won't exist until generated). Add a `prepare` or `pretest` script to `package.json` that runs the build step automatically. Document this in README.

## Risks

- **Build step complexity** -- template markers and build script become another thing to maintain and debug when adding/editing skills
  - Likelihood: medium
  - Impact: medium
  - Mitigation: Keep template syntax minimal (simple variable substitution, not a full template engine). Document the marker format. Validate output in CI.

- **Divergence creep** -- over time, Claude and Codex variants may need genuinely different logic (not just path/syntax swaps), making templates inadequate
  - Likelihood: low
  - Impact: high
  - Mitigation: Design the template system with escape hatches (conditional blocks for Agent()-vs-prose spawn blocks). If divergence grows beyond templates, allow per-platform override files that bypass the template entirely.

- **Install breakage** -- consolidated install could silently break one platform while the other works
  - Likelihood: medium
  - Impact: high
  - Mitigation: Phase 00 captures a v1.19.0 baseline snapshot of both install outputs. Every subsequent phase runs a diff against this baseline before being considered complete. The baseline script records: (a) the complete file listing produced by each install function, (b) the content hash of every installed file. This makes regressions immediately visible.

- **Accidental divergence loss** -- template markers could fail to capture a real platform-specific difference, producing broken output for one platform
  - Likelihood: medium
  - Impact: high
  - Mitigation: Phase 00 audits every file pair and classifies divergences line-by-line. The audit output drives template marker placement in Phase 01. No file is templated without understanding its specific divergence pattern first.

## Phase Plan

- **Phase 00 -- audit-and-baseline**: Foundation phase. Four deliverables:
  1. **File pair audit** -- Script that compares every file pair (references, skills, agents) and classifies each as `identical` (byte-equal, no template needed) or `divergent` (with a line-by-line diff summary showing which divergence patterns apply: path substitution, agent name prefix, Agent() vs prose spawn, prose wording). Output: `audit-report.md` in project folder. Current counts: 22 identical reference pairs + 25 divergent reference pairs, 1 identical skill pair + 12 divergent skill pairs, 0 identical agent pairs + 8 divergent agent pairs.
  2. **install.cjs testability** -- Make `bin/install.cjs` import-safe by wrapping ALL side effects (early-exit guards like the `skills/do/` existence check, the interactive TTY prompt block, and direct `process.exit()` calls) inside a `require.main === module` guard. When `require()`d, the file exports `{ installClaudeCode, installCodex }` without executing any prompts or exiting the process. When run directly (e.g., via `node bin/install.cjs` or as postinstall), it behaves exactly as today. This is a minimal, backward-compatible change that enables the baseline script to import and call the install functions programmatically.
  3. **v1.19.0 baseline snapshot** -- Script that `require()`s `install.cjs` (using the new exports from step 2), stubs `os.homedir()` to point to a temp directory, creates the necessary directory stubs (`tempHome/.claude` and `tempHome/.codex`) so install functions don't early-return, calls `installClaudeCode()` and `installCodex()`, and records: (a) complete file listing with relative paths, (b) SHA-256 hash of every installed file. Output: `baseline-claude.json` and `baseline-codex.json` in project folder. These are the regression targets for all subsequent phases. Important: baseline must be captured BEFORE any files are moved to `skills/_source/`, while both `skills/do/` and `skills/codex/` still contain hand-maintained originals.
  4. **Template marker design** -- Based on audit findings, define the template syntax. Important: the delimiter must NOT conflict with existing `{{...}}` Mustache/Handlebars-style placeholders already used in scaffold templates (e.g., `{{TITLE}}`, `{{VISION}}` in reference files). Use ERB-style delimiters: `<%= PLATFORM_PATH %>` (variable substitution), `<%= AGENT_PREFIX %>` (agent name prefix), `<% if claude %>...<% endif %>` / `<% if codex %>...<% endif %>` (conditional blocks for Agent() vs prose spawn and prose wording differences). These are visually distinct from existing `{{...}}` content and won't trigger false positives in validation. Document in a `template-spec.md` in the project folder.

- **Phase 01 -- references-dedup**: Single-source the 47 reference files. Three sub-steps:
  1. **Create `skills/_source/references/`** -- Move the 47 reference files from `skills/do/references/` to `skills/_source/references/`. For the 22 identical pairs, these files need no template markers. For the 25 divergent pairs, add template markers based on the audit findings from Phase 00. Delete hand-maintained `skills/codex/references/` entirely.
  2. **Implement build script** -- Create `scripts/build-skills.cjs` that reads `skills/_source/references/`, applies template substitution for both platforms, and writes resolved output to `skills/do/references/` (Claude variant) and `skills/codex/references/` (Codex variant). For identical files, both outputs are byte-equal. For divergent files, each output has platform-specific values. Add `prepack` script to `package.json` that runs the build. The build script must validate output contains no unresolved ERB-style markers (fail on any `<%= ` or `<% if ` remaining in generated files). Also configure npm packaging rules: add `.npmignore` (or `package.json files[]`) to exclude `skills/_source/` and `agents/_source/` from the tarball while including generated `skills/do/`, `skills/codex/`, and `agents/`. Verify with `npm pack --dry-run`.
  3. **Verify** -- Run both install functions (using the exports from Phase 00) and diff output against v1.19.0 baseline snapshot. Both platforms must produce bit-identical results to the baseline.

- **Phase 02 -- skills-agents-dedup** (MVP): Single-source the 13 entry skills + 8 agents. Three sub-steps:
  1. **Entry skills (12 divergent + 1 identical)** -- Move entry-level skill markdown files from `skills/do/*.md` to `skills/_source/*.md`. Add template markers for divergent files. Extend the build script to process entry skills, generating both `skills/do/*.md` (Claude) and `skills/codex/*.md` (Codex). The build script must handle Agent() JS blocks vs natural-language spawn directives using conditional blocks. Delete hand-maintained `skills/codex/*.md` files. Note: `scripts/` directory lives in `skills/_source/scripts/` and is copied to `skills/do/scripts/` by the build step as-is (no template processing). The build step does NOT copy scripts into `skills/codex/`. `install.cjs` already copies `skills/do/scripts/` into the Codex install target explicitly (line 79 of current `install.cjs`). This behavior is preserved.
  2. **Agents (8 divergent pairs)** -- Move agent markdown files to `agents/_source/` (using a neutral naming convention, e.g., `_source/code-reviewer.md`). Add template markers. Extend the build script to generate `agents/do-*.md` and `agents/codex-*.md` from the canonical source files (with filename prefix rewriting). Delete hand-maintained agent files from `agents/`.
  3. **install.cjs review** -- Verify `install.cjs` still works correctly with generated files. The `installCodex()` function reads from `skills/codex/` (now generated) and copies `do-` prefixed scripts from `skills/do/scripts/`. Agent installation filters on `codex-` prefix for Codex target. No structural changes expected but must be verified against baseline.
  Verify: diff both install outputs against v1.19.0 baseline snapshot.

- **Phase 03 -- polish**: Quality pass on canonical sources. Add `.gitignore` entries for generated output directories (`skills/do/`, `skills/codex/`, `agents/do-*.md`, `agents/codex-*.md`) since they are build artifacts -- only `skills/_source/` and `agents/_source/` are committed. Verify tarball contents via `npm pack --dry-run` to confirm generated output IS included (gitignored but not npmignored) and `_source/` dirs are excluded from the tarball (consumers don't need templates). User runs `/skill-creator` per skill in `skills/_source/`. README + project.md update. Version bump + release.

Phases are strictly sequential -- each depends on the previous. MVP is reached at the end of Phase 02, when all platform-specific entry skills and agents are consolidated alongside the references.
