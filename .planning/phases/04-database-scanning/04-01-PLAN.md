Create a CommonJS Node.js script that analyzes a project directory and returns structured JSON.

**Per D-01: Parse package.json AND requirements.txt for tech detection**

For JavaScript/TypeScript projects (package.json exists):
- Read package.json, merge dependencies + devDependencies
- Detect frameworks: react, vue, next, @angular/core, svelte, vite, webpack
- Detect UI libraries: @mui/material, @chakra-ui/react, tailwindcss, @mantine/core, antd
- Detect testing: vitest, jest, mocha, @testing-library/react, cypress, playwright
- Detect linting: eslint, prettier, biome
- Detect state: @reduxjs/toolkit, zustand, jotai, recoil, mobx
- Detect routing: @tanstack/react-router, react-router, next (implies routing)
- Detect forms: react-hook-form, formik, zod, yup

For Python projects (requirements.txt exists, no package.json):
- Read requirements.txt, parse lines (ignore comments #, version specifiers)
- Detect frameworks: django, flask, fastapi, tornado, pyramid
- Detect testing: pytest, unittest, nose, tox
- Detect linting: flake8, pylint, black, mypy, ruff
- Detect data: pandas, numpy, scipy, polars
- Detect web: requests, httpx, aiohttp
- Detect ORM: sqlalchemy, django (implies ORM), peewee, tortoise-orm

**Monorepo detection (per council advisory):**
- Check for monorepo indicators: lerna.json, pnpm-workspace.yaml, rush.json, nx.json, turbo.json
- If found, add to output: `"monorepo": { "type": "lerna|pnpm|rush|nx|turbo", "warning": "Consider running /do:scan from individual package directories" }`

**Per D-02: Scan src/ for non-obvious directories**
- Read src/ top-level directories (or project root if no src/)
- Skip obvious: components, pages, utils, lib, assets, styles, types
- Include non-obvious directories with inferred purpose
- Infer purpose from name: api/services -> "API layer", hooks -> "Custom hooks", store/redux -> "State management", routes -> "Routing", i18n/locales -> "Internationalization", common -> "Shared code", features -> "Feature modules"

**Per D-03: Detect config files**
- Check existence of: .eslintrc.*, eslint.config.*, tsconfig.json, vite.config.*, webpack.config.*, commitlint.config.*, prettier.config.*, .prettierrc.*, pyproject.toml, setup.py, setup.cfg
- For each found, note implications (e.g., commitlint.config -> "Conventional commits enforced")

**Per D-04: Read recent commits for prefix patterns**
- Check if git repo: execSync('git rev-parse --git-dir')
- Get last 50 commits: execSync('git log --oneline -50 --format="%s"')
- Match conventional commit pattern: /^(feat|fix|chore|docs|style|refactor|perf|test|build|ci|revert)(\(.+\))?:/
- Return found prefixes and pattern type
- If no conventional commits detected, return empty array (do NOT fabricate patterns)

**Output JSON structure:**
```json
{
  "project_name": "string (from package.json name or folder name)",
  "project_type": "javascript|python|unknown",
  "detected": {
    "frameworks": ["react", "vite"],
    "ui_libraries": ["@mui/material"],
    "testing": ["vitest", "@testing-library/react"],
    "linting": ["eslint", "prettier"],
    "state_management": ["@reduxjs/toolkit"],
    "routing": ["@tanstack/react-router"],
    "forms": ["react-hook-form", "zod"],
    "data": [],
    "orm": []
  },
  "key_directories": {
    "src/api/": "API layer",
    "src/store/": "State management"
  },
  "conventions": {
    "commit_prefixes": ["feat", "fix", "chore"],
    "commit_pattern": "conventional-commits"
  },
  "config_files": [
    { "file": "vite.config.ts", "implications": "Vite build tool" }
  ],
  "monorepo": null
}
```

**CLI interface (follow project-health.cjs pattern):**

- `node scan-project.cjs <project-path> --pretty` - pretty-printed JSON
- `node scan-project.cjs --help` - help text
- Export `scanProject(projectPath)` for programmatic use

**Error handling:**
- If package.json missing/invalid AND requirements.txt missing: continue with empty detected, use folder name for project_name, set project_type to "unknown"
- If not a git repo: skip commit detection, return empty conventions (do NOT fabricate branch naming patterns)
- If src/ missing: return empty key_directories

**component-readme.md:**
```markdown
# Components

Documentation for shared/complex components in this project.

## When to Document

Document a component when:
- It has non-obvious props or configuration
- It wraps external libraries with custom logic
- It's used widely and has conventions for usage
- It has domain-specific behavior (e.g., FormFields that know about API types)

## Format

Each component doc should include:
- **Purpose**: What the component does
- **Props**: Key props and their types
- **Usage**: Example code showing common patterns
- **Gotchas**: Things that aren't obvious

## Example Entry

See leaselinq-frontend/components/FormFields.md for a good example of documenting form field wrappers.
```

**tech-readme.md:**
```markdown
# Tech

Documentation for technical patterns, utilities, and architectural decisions.

## When to Document

Document a pattern when:
- It's project-specific and not obvious from the code
- It involves multiple files working together
- Future developers would waste time figuring it out
- It wraps external libraries with custom conventions

## Format

Each tech doc should include:
- **What**: Brief description of the pattern
- **When**: When to use this pattern
- **How**: Code examples and file references
- **Why**: Rationale for this approach

## Example Topics

- API layer & code generation
- Form system patterns
- State management conventions
- Routing & layout patterns
```

**features-readme.md:**
```markdown
# Features

Documentation for complex features spanning multiple components/files.

## When to Document

Document a feature when:
- It spans multiple routes, components, or API endpoints
- It has business logic that isn't obvious from the code
- It went through significant iterations worth capturing
- It integrates with external systems

## Format

Each feature doc should include:
- **Overview**: What the feature does for users
- **Architecture**: How the pieces fit together
- **Key Files**: Which files implement the feature
- **API Endpoints**: Backend endpoints used (if applicable)
- **Known Issues**: Current limitations or tech debt

## Example Entry

See leaselinq-frontend/features/documents-emails.md for a feature doc example.
```
