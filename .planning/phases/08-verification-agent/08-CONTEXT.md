# Phase 8: Verification Agent - Context

**Gathered:** 2026-04-13
**Status:** Ready for planning

<domain>
## Phase Boundary

Implement the verification stage that runs after execution completes. Verification compares the implementation against the full task context, runs applicable quality checks, and gates task completion on passing results plus user UAT approval.

**Delivers:**
- `references/stage-verify.md` — verification flow
- Checklist-style verification against full task context
- Quality check auto-detection from package.json
- Pass/Fail result handling with completion blocking
- UAT approval flow with generated checklist
- Context-aware handoff when context is full

</domain>

<decisions>
## Implementation Decisions

### Verification Scope
- **D-23:** Compare against full task context — Problem Statement, Approach, Concerns, and Clarifications. Not just Approach alone.
- **D-24:** Checklist-style verification — parse Approach section into discrete steps, mark each as done/not done. Creates clear audit trail.

### Quality Checks
- **D-25:** Auto-detect quality checks from package.json — look for `lint`, `typecheck`/`tsc`, `test` in scripts. Run what exists.
- **D-26:** Block completion on quality check failures — task cannot be marked complete until all checks pass.
- **D-27:** On quality failure, offer two options: (1) spawn new `/do:task` to fix the failing check, or (2) user handles manually. User chooses.

### Issue Handling
- **D-28:** Binary Pass/Fail result — no warnings or partial states. Either verification passes entirely or it fails.

### Completion Flow
- **D-29:** Two-stage completion — verification pass sets `stage: verified`, then UAT approval sets `stage: complete`.
- **D-30:** UAT checklist generated by Claude at verification time — based on what was actually implemented, create user-verifiable items (e.g., "Check login form shows validation error").
- **D-31:** On UAT approval — set `stage: complete`, set `stages.verification: complete`, clear `active_task` from config.json.

### Context-Aware Handoff
- **D-32:** On UAT failure, check context usage:
  - If context < 80%: offer to loop back to execution or spawn fix inline (Claude's discretion based on scope)
  - If context >= 80%: document current state in task markdown, generate ready-to-paste handoff prompt containing database folder, github projects folder, task ID/location, relevant context, and instruction to start a new `/do:task`

### Claude's Discretion
- Exact quality check script detection logic (what counts as lint, typecheck, test)
- Format and content of UAT checklist items
- How to detect context usage percentage
- Whether to loop back or spawn new task when context is OK but UAT fails

</decisions>

<specifics>
## Specific Ideas

- Verification Results section in task markdown — append after Execution Log:
  ```markdown
  ## Verification Results
  
  ### Approach Checklist
  - [x] Step 1 description
  - [x] Step 2 description
  - [ ] Step 3 description (INCOMPLETE: reason)
  
  ### Quality Checks
  - Lint: PASS
  - Types: PASS
  - Tests: FAIL (3 failing tests)
  
  ### Result: FAIL
  ```

- UAT checklist example:
  ```
  Please verify manually:
  1. [ ] Login form shows validation error for invalid email
  2. [ ] Submit button is disabled while loading
  3. [ ] Success redirects to dashboard
  
  Mark complete? (yes/no)
  ```

- Handoff prompt template for context >= 80%:
  ```
  Database: ~/workspace/database
  Project: ~/workspace/github-projects/<project>
  Task: .do/tasks/<task-id>.md
  
  UAT failed. Context at 80%+. Please start fresh:
  /do:task "Fix: <summary of what failed UAT>"
  
  Prior context: <brief summary of what was done and what failed>
  ```

</specifics>
