# Phase 8: Verification Agent - Research

**Researched:** 2026-04-13
**Domain:** Post-implementation verification, quality checks, UAT flow
**Confidence:** HIGH

## Summary

This phase implements the verification stage that runs after task execution completes. The verification agent compares implementation against the full task context (Problem Statement, Approach, Concerns, Clarifications), auto-detects and runs quality checks from package.json, generates a UAT checklist for user approval, and handles completion flow including context-aware handoff when context is near capacity.

The implementation follows the established stage reference file pattern (`stage-verify.md`), consistent with `stage-grill.md` and `stage-execute.md`. Key technical challenges are: (1) parsing the Approach section into discrete verifiable steps, (2) detecting quality check scripts from package.json with varying naming conventions, and (3) estimating context usage percentage to decide between inline fix vs handoff.

**Primary recommendation:** Create `stage-verify.md` following the existing stage reference pattern. Use package.json script detection for quality checks (look for `lint*`, `typecheck*`/`tsc*`, `test*` patterns). For context usage detection, use a heuristic based on conversation length since direct programmatic access to `context_window.used_percentage` is not available from within a skill.

### Pattern 3: Quality Check Detection
**What:** Auto-detect quality check scripts from package.json
**When to use:** Step V3 of verification

**Detection rules (per D-25):**
| Script Pattern | Quality Check Type | Examples |
|----------------|-------------------|----------|
| `lint*` | Linting | `lint`, `lint:fix`, `lint:check` |
| `typecheck*`, `tsc*`, `type-check*` | Type checking | `typecheck`, `tsc`, `type-check` |
| `test*` (excluding `test:*:watch`) | Testing | `test`, `test:unit`, `test:integration` |

**Example detection code:**
```javascript
function detectQualityChecks(packageJson) {
  const scripts = packageJson.scripts || {};
  const checks = [];
  
  for (const [name, cmd] of Object.entries(scripts)) {
    if (/^lint/i.test(name)) {
      checks.push({ type: 'lint', script: name, command: cmd });
    } else if (/^(typecheck|tsc|type-check)/i.test(name)) {
      checks.push({ type: 'types', script: name, command: cmd });
    } else if (/^test/i.test(name) && !/watch/i.test(name)) {
      checks.push({ type: 'test', script: name, command: cmd });
    }
  }
  
  return checks;
}
```

### Pattern 4: UAT Checklist Generation
**What:** Generate user-verifiable checklist based on actual implementation
**When to use:** After quality checks pass (Step V5)

**Example checklist:**
```
Please verify manually:
1. [ ] Login form shows validation error for invalid email
2. [ ] Submit button is disabled while loading
3. [ ] Success redirects to dashboard
4. [ ] Error toast appears on API failure

Mark complete? (yes/no)
```

**Generation approach:** Based on the Approach section and Execution Log, identify user-observable behaviors. Focus on:
- UI state changes
- Error handling paths
- Success paths
- Edge cases mentioned in Concerns

### Anti-Patterns to Avoid
- **Assuming quality scripts exist:** Always check package.json first; not all projects have lint/test scripts
- **Running all test suites:** Use focused test runs if available (e.g., `test:unit` vs full `test`)
- **Blocking on slow tests:** Consider timeout handling for long-running test suites
- **Hardcoding script names:** Use pattern matching, not exact matches like `npm run lint`

## Common Pitfalls

### Pitfall 1: Quality Check Script Variants
**What goes wrong:** Missing quality checks because script names vary across projects (`lint` vs `eslint` vs `lint:check`)
**Why it happens:** No standardized naming convention in npm ecosystem
**How to avoid:** Use pattern matching (`/^lint/i`) not exact matches (`=== 'lint'`)
**Warning signs:** Verification passes but obvious lint errors exist in changed files

### Pitfall 2: Test Timeout Handling
**What goes wrong:** Verification hangs indefinitely waiting for slow test suites
**Why it happens:** Some test suites take minutes to run
**How to avoid:** Consider adding a timeout (e.g., 5 minutes) with graceful failure message
**Warning signs:** User waiting more than 2-3 minutes with no feedback

### Pitfall 3: Context Percentage Detection
**What goes wrong:** Trying to access `context_window.used_percentage` programmatically
**Why it happens:** This data is only available via statusline JSON (external script) or `/context` command output
**How to avoid:** Use a heuristic: count conversation turns, estimate tokens from Execution Log size, or use a fixed threshold (e.g., if Execution Log > 10 entries, assume context is heavy)
**Warning signs:** Attempting to read session.json or other Claude internal files

### Pitfall 4: Handoff Prompt Completeness
**What goes wrong:** Handoff prompt missing critical context for fresh session
**Why it happens:** Assuming the new session will have access to same files
**How to avoid:** Include absolute paths, task ID, specific failure reason, and relevant context summary
**Warning signs:** User in new session has to re-read everything to understand the task

### Pitfall 5: Stage Transition Atomicity
**What goes wrong:** Partial state updates leave task in inconsistent state
**Why it happens:** Updating `stage:` but forgetting `stages.X:` field
**How to avoid:** Always update both `stage:` AND `stages.<stage>:` together (per D-22 from Phase 7)
**Warning signs:** `/do:continue` routes to wrong stage handler

### Context Usage Heuristic
Since direct access to `context_window.used_percentage` is not available from within a skill, use this heuristic:

```javascript
function estimateContextUsage(taskMarkdown) {
  // Rough estimation based on observable factors
  const executionLogEntries = (taskMarkdown.match(/^### \d{4}-\d{2}-\d{2}/gm) || []).length;
  const clarificationCount = (taskMarkdown.match(/^\*\*Q:\*\*/gm) || []).length;
  const approachSteps = (taskMarkdown.match(/^\d+\./gm) || []).length;
  
  // Each execution log entry ~500-1000 tokens
  // Each Q&A pair ~200-400 tokens
  // Each approach step ~100-200 tokens (read multiple times)
  
  const estimatedTokens = 
    (executionLogEntries * 750) + 
    (clarificationCount * 300) + 
    (approachSteps * 150);
  
  // Assume 200k context window, reserve 40k buffer
  const usableContext = 160000;
  
  return Math.min(100, Math.round((estimatedTokens / usableContext) * 100));
}

// Usage
const estimated = estimateContextUsage(taskMarkdown);
if (estimated >= 80) {
  // Generate handoff prompt
} else {
  // Offer inline fix options
}
```

**Note:** This is a rough heuristic. The actual implementation may use simpler logic like counting entries or checking file size. The key is having SOME mechanism to decide between inline vs handoff, per D-32.

## Open Questions

1. **Exact context percentage threshold**
   - What we know: D-32 specifies 80% threshold for handoff decision
   - What's unclear: How to reliably estimate context percentage without direct API access
   - Recommendation: Use file-size or entry-count heuristic; err on side of offering handoff earlier rather than later

2. **Quality check timeout**
   - What we know: Some test suites can run for several minutes
   - What's unclear: What's an acceptable timeout before declaring failure?
   - Recommendation: 5-minute default timeout with clear messaging; user can re-run manually if needed

3. **Verification of wave-based tasks**
   - What we know: Tasks can have waves (per Phase 5 D-02)
   - What's unclear: Should verification run per-wave or only at task completion?
   - Recommendation: Verify at task completion (after all waves), keep it simple for v1
